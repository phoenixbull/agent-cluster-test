## 时间线对比

### 传统开发流程
```
需求 → 开发 (4h) → 自测 (1h) → Code Review (1h) → 修改 (1h) → 测试 (2h) → 部署 (1h)
总计：10-12 小时
```

### v2.0 AI Agent 流程
```
需求 → Zoe 拆解 (5min) → Agent 开发 (30min) → AI Review + CI (15min) → 人工 Review (5min) → 合并
总计：1-2 小时 (人工投入仅 10 分钟)
```

## 真实案例：企业客户自定义模板功能

### 背景
一个企业客户打电话来，说希望能复用他们已经配置好的设置，在团队内共享。

### 完整流程

#### 第 1 步：Zoe 理解并拆解 (5 分钟)
```
通话结束后，作者和 Zoe 聊了聊这个需求。

神奇之处：零解释成本！
因为所有会议记录自动同步到 Obsidian，Zoe 已经读过了通话内容，
知道客户是谁、他们的业务场景、现有配置。
```

**Zoe 拆解结果：**
- 做一个模板系统，让用户保存和编辑现有配置
- 给客户充值 — 用管理员 API 立即解除客户的使用限制
- 拉取客户配置 — 从生产数据库（只读）获取客户现有的设置
- 生成 prompt 并启动 Codex Agent

#### 第 2 步：创建隔离环境 (2 分钟)
```bash
# 创建 worktree + 启动 Agent
git worktree add ../feat-custom-templates -b feat/custom-templates
cd ../feat-custom-templates && pnpm install

# 启动 tmux 会话
tmux new-session -d -s "codex-templates" \
  -c "/workspace/feat-custom-templates" \
  "./run-agent.sh templates gpt-5.3-codex high"
```

#### 第 3-6 步：Agent 执行 + Review + CI (30 分钟)
```
Codex Agent 在后台运行：
• 分析代码库
• 实现模板功能
• 编写测试
• 提交代码
• 创建 PR #341

同时：
• 3 个 AI Reviewer 自动审查
• CI 管道自动运行
```

#### 第 7 步：监控 (自动)
```
Cron 每 10 分钟检查：
✅ tmux 会话存活
✅ PR 已创建 (#341)
✅ CI 全绿
✅ Codex Reviewer 批准
✅ Gemini Reviewer 批准
✅ UI 截图已附
```

#### 第 8 步：Telegram 通知
```
🎉 PR #341 准备好了，可以 Review！

任务：实现自定义邮件模板功能
Agent: codex
分支：feat/custom-templates

✅ CI 全绿
✅ Codex Reviewer 批准
✅ Gemini Reviewer 批准
📸 UI 截图已附

https://github.com/xxx/pull/341
```

#### 第 9 步：人工 Review (5 分钟)
```
作者的 Review：
• 看截图 → 功能正确 ✓
• 看 AI Reviewer 评论 → 边界情况已处理 ✓
• 直接合并！
```

#### 第 10-11 步：合并 + 部署 (3 分钟)
```
• Squash and Merge
• 自动部署到生产环境
• 通知客户功能已上线
```

### 总结
- **总耗时**: 1-2 小时
- **人工投入**: 10 分钟
- **客户体验**: 当天提需求，当天看到效果
- **转化**: 直接变成付费用户

## Ralph Loop 学习机制

### 失败处理流程

```
Agent 执行失败
       │
       ▼
┌─────────────────┐
│ 分析失败原因     │
│ • CI 失败？       │
│ • Review 拒绝？   │
│ • 超时？         │
└─────────────────┘
       │
       ▼
┌─────────────────┐
│ 检索历史记忆     │
│ • 类似任务成功   │
│ • 类似任务失败   │
│ • 成功模式      │
│ • 失败模式      │
└─────────────────┘
       │
       ▼
┌─────────────────┐
│ 动态调整 Prompt  │
│ ❌ "实现模板功能" │
│ ✅ "停。客户要的  │
│    是配置复用，   │
│    不是新建。    │
│    这是会议原话： │
│    '我们希望...'" │
└─────────────────┘
       │
       ▼
┌─────────────────┐
│ 重试 (最多 3 次)   │
└─────────────────┘
       │
       ├────── 成功 ──────► 保存成功模式
       │
       └────── 失败 ──────► 保存失败模式
                             │
                             ▼
                       通知人工介入
```

### 学习成果示例

**成功模式被记录：**

## Agent 选择策略

### 任务类型映射

```python
TASK_TYPE_MAPPING = {
    # 后端任务 → Codex (90% 的任务)
    "backend_logic": "codex",
    "bug_fix": "codex",
    "refactoring": "codex",
    "api_design": "codex",
    "database": "codex",
    
    # 前端任务 → Claude Code
    "frontend": "claude-code",
    "ui_component": "claude-code",
    "git_operation": "claude-code",
    "quick_fix": "claude-code",
    
    # 设计任务 → Designer (Gemini)
    "ui_design": "designer",
    "visual": "designer",
    "html_css": "designer",
    "design_spec": "designer",
    
    # 多 Agent 协作
    "full_feature": ["designer", "claude-code", "codex"],
    "new_page": ["designer", "claude-code"],
}
```

### 选择示例

| 任务描述 | 选中 Agent | 原因 |
|----------|------------|------|
| "修复登录 API 的 bug" | codex | 后端逻辑问题 |
| "首页按钮样式调整" | claude-code | 前端快速迭代 |
| "设计新的仪表盘" | designer | UI 设计任务 |
| "实现完整的用户管理功能" | designer → claude-code → codex | 多 Agent 协作 |

## 监控与通知

### Cron 监控 (每 10 分钟)

```python
检查项：
├─ tmux 会话是否存活
├─ PR 是否创建
├─ CI 状态 (lint/typecheck/tests/e2e)
├─ Review 状态 (3 个 AI Reviewer)
└─ 是否需要重试/人工介入
```

### 通知事件

| 事件 | 通知渠道 | 内容 |
|------|----------|------|
| task_complete | Telegram | PR 已就绪，可以 Review |
| task_failed | Telegram | 多次重试失败，需要人工介入 |
| pr_ready | Telegram | PR 已创建，等待 Review |
| human_intervention | Telegram | 需要人工决策 |

## 完成定义

PR 算"完成"必须满足：

- ✅ **PR 已创建**
- ✅ **分支已同步到 main** (没有冲突)
- ✅ **CI 通过** (lint、typecheck、tests、e2e)
- ✅ **Codex Reviewer 批准**
- ✅ **Gemini Reviewer 批准**
- ✅ **UI 改动附带截图**

只有全部满足，才算真正完成！

---

## 关键成功因素

1. **上下文分离**: 编排层持有业务上下文，执行层只拿最小必要信息
2. **动态 Prompt**: 失败时分析原因并调整 prompt，不是简单重试
3. **多 Reviewer**: 3 个 AI 审查者，各有侧重
4. **完成定义**: PR+CI+Review+截图，全部满足才算完成
5. **人工介入点**: 只在最终 Review 需要人工 (5-10 分钟)
