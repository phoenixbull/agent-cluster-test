## 📋 标准 Bug 报告格式

### 完整 Bug 报告

```markdown
# 🔧 Bug 修复任务

## 📋 Bug 信息

**Bug ID**: `BUG-20260309213000`
**严重程度**: 🔴 CRITICAL
**Bug 类型**: 安全问题
**标题**: 用户认证绕过漏洞

---

## 📝 Bug 描述

在用户认证模块中发现严重安全漏洞，攻击者可以通过构造特殊请求绕过认证直接访问受保护的资源。

---

## 🔄 重现步骤

1. 启动应用
2. 不登录，直接访问 `/api/v1/posts`
3. 使用 curl 发送请求：
   ```bash
   curl -H "Authorization: Bypass" http://localhost:8000/api/v1/posts
   ```
4. 观察到返回了所有文章数据（未认证应该返回 401）

---

## ✅ 期望行为

- 未认证用户访问受保护资源应返回 `401 Unauthorized`
- 只有携带有效 JWT Token 的请求才能访问

---

## ❌ 实际行为

- 未认证用户可以访问所有资源
- 认证检查被绕过

---

## 📍 代码位置

**文件**: `backend/app/api/posts.py`

**行号**: 25-30

**错误信息**:
```
WARNING: Authentication bypass detected
No valid token check in get_posts endpoint
```

---

## 🛠️ 修复要求

### 必须完成

1. **修复 Bug** - 添加 JWT Token 验证
2. **添加测试** - 添加认证测试用例
3. **验证修复** - 确保未认证请求返回 401

### 修复建议

**安全相关修复建议**:
- 使用 `get_current_user` 依赖注入进行认证
- 不要手动检查 Token，使用框架提供的装饰器
- 对所有 API 端点统一实施认证检查

**代码示例**:
```python
from app.utils.dependencies import get_current_user

@router.get("/posts")
async def get_posts(
    current_user: User = Depends(get_current_user),  # ← 添加认证
    db: Session = Depends(get_db)
):
    # 现在只有认证用户才能访问
    posts = db.query(Post).all()
    return posts
```

## 🎯 提示词最佳实践

### 好的提示词特点

✅ **具体明确**
- 明确指出 Bug 位置（文件 + 行号）
- 提供完整的错误信息
- 给出期望和实际行为对比

✅ **可重现**
- 提供详细的重现步骤
- 包含测试用例或 curl 命令
- 说明触发条件

✅ **有修复建议**
- 提供修复方向
- 给出代码示例（如适用）
- 说明注意事项

✅ **有验收标准**
- 明确修复完成的标准
- 说明需要添加的测试
- 列出提交要求

### 避免的问题

❌ **模糊不清**
- "这里有个 Bug" → 哪里？什么 Bug？
- "修复一下" → 怎么修复？修复成什么样？

❌ **缺少上下文**
- 只给错误信息，不给代码位置
- 只给现象，不给重现步骤

❌ **没有验收标准**
- 修复后如何验证？
- 需要添加哪些测试？
