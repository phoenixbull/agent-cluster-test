# 2026-03-04 - Agent 集群系统 v2.0 升级

## 升级来源

基于 [@elvissun 的 OpenClaw + Codex 实践](https://x.com/elvissun/status/2025920521871716562)

**核心成果**:
- 单日最高 94 次提交（平均每天 50 次）
- 30 分钟完成 7 个 PR
- 当天交付客户需求
- 月成本 $190（Claude $100 + Codex $90）

## 核心架构变化

### v1.0 → v2.0

| 特性 | v1.0 | v2.0 |
|------|------|------|
| 架构 | 单层 Agent | 双层（编排层 + 执行层） |
| 上下文 | 分散 | 集中管理（Zoe 持有） |
| 失败处理 | 简单重试 | Ralph Loop 动态调整 prompt |
| Agent 选择 | 手动 | 自动策略 |
| 监控 | 基础 | 每 10 分钟自动检查 |
| Code Review | 无 | 3 个 AI Reviewer |
| 完成定义 | PR 创建 | PR+CI+Review+截图 |

## 新增组件

### 1. 编排层 (Zoe)
- 模型：qwen-max
- 职责：任务调度、上下文管理、Prompt 生成、失败分析

### 2. 执行层 Agent
- **Codex** (qwen-coder-plus): 后端逻辑、复杂 bug、多文件重构
- **Claude Code** (qwen-plus): 前端工作、git 操作、快速迭代
- **Gemini** (qwen-vl-plus): UI 设计、视觉规范

### 3. 审查层
- Codex Reviewer (qwen-coder-plus): 边界情况、逻辑错误
- Gemini Reviewer (qwen-plus): 安全问题、扩展性
- Claude Reviewer (qwen-turbo): 基础检查

### 4. 核心机制
- **Ralph Loop**: 失败时动态调整 prompt，不是简单重试
- **Agent 选择策略**: 根据任务类型自动选择
- **自动化监控**: 每 10 分钟检查 tmux/PR/CI/Review 状态

## 完整工作流

```
客户需求 → Zoe 理解拆解 → 创建 worktree+tmux → 启动 Agent
                                                      ↓
人工 Review ← Telegram 通知 ← CI+Review 通过 ← Agent 创建 PR
    ↓
  合并
```

## 关键设计

1. **上下文分离**: 编排层持有业务上下文，执行层只拿最小必要信息
2. **动态 Prompt**: 失败时分析原因并调整 prompt
3. **多 Reviewer**: 3 个 AI 审查者各有侧重
4. **完成定义**: PR+CI+Review+截图全部满足才算完成
5. **人工介入点**: 只在最终 Review 需要人工
